# ADR-0004 — CQRS with separate read models on the write stream

**Status:** Accepted

## Context
The ledger's write side must optimize for correct, ordered, append-only commits under
concurrency; its read side must optimize for low-latency queries (balances, statements,
transfer status) at much higher volume. A single normalized model forces a compromise that
serves neither well, and querying the raw event store for every read is prohibitively slow.

## Decision
Apply **CQRS**: commands append events to the event store (write model); **projection
builders** fold those events into denormalized **read models**, one per query shape
(`account_balance_view`, `transaction_history_view`, `transfer_status_view`). Queries hit
read models (and a Redis cache for hot balances), never the event store.

## Consequences
- **+** Read and write scale independently; reads can be replicated and cached freely.
- **+** New query needs → new projection, built by replaying existing events; no write-model
  change and no data migration.
- **+** A projection bug is fixed by replaying the stream into a new projection version and
  swapping — data self-heals from the source of truth.
- **−** **Eventual consistency**: a read may briefly lag a just-committed write. Handled with
  read-your-writes via the `transfer_status_view` and cache invalidation; acceptable because
  the authoritative balance always lives in the event store.
- **−** More moving parts (consumers, offsets, rebuild tooling). Cost accepted for scalability.
